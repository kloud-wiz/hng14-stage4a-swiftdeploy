# SwiftDeploy

A declarative CLI tool that generates and manages a containerized service stack from a single `manifest.yaml`. Nothing is handwritten — the manifest is the single source of truth.

Built in two stages:
- **Stage 4A** — Core engine: manifest-driven config generation, container lifecycle management
- **Stage 4B** — Observability & policy enforcement: Prometheus metrics, OPA policy gating, live status dashboard, audit reporting

## Project Structure

```
.
├── manifest.yaml              # Single source of truth — only file you edit
├── swiftdeploy                # CLI executable
├── Dockerfile                 # Builds the API service image
├── app/
│   ├── main.py                # Python Flask API service
│   └── requirements.txt       # App dependencies
├── templates/
│   ├── nginx.conf.j2          # Nginx config template
│   └── docker-compose.yml.j2  # Docker Compose template
├── policies/
│   ├── infrastructure.rego    # Disk/CPU/memory policy
│   ├── canary.rego            # Error rate/latency policy
│   └── data.json              # Policy threshold values
├── nginx.conf                 # Generated by swiftdeploy init
├── docker-compose.yml         # Generated by swiftdeploy init
├── history.jsonl              # Generated by swiftdeploy status
├── audit_report.md            # Generated by swiftdeploy audit
└── requirements.txt           # CLI dependencies
```

## Prerequisites

- Docker
- Docker Compose plugin (`docker compose`)
- Python 3.10+

## Setup

**1. Clone the repo:**
```bash
git clone https://github.com/kloud-wiz/hng14-stage4a-swiftdeploy.git
cd hng14-stage4a-swiftdeploy
```

**2. Create and activate a virtual environment:**
```bash
python3 -m venv venv
source venv/bin/activate
```

> **Note:** You must activate the virtual environment every time you open a new terminal session before running `./swiftdeploy`. If you see import errors, run `source venv/bin/activate` first.

**3. Install CLI dependencies:**
```bash
pip install -r requirements.txt
```

**4. Build the Docker image:**
```bash
docker build -t kloudwiz-swiftdeploy:latest .
```

**5. Make the CLI executable:**
```bash
chmod +x swiftdeploy
```

## Subcommand Walkthrough

### `init`
Parses `manifest.yaml` and generates `nginx.conf` and `docker-compose.yml` in the project root.
```bash
./swiftdeploy init
```

### `validate`
Runs 5 pre-flight checks and reports pass/fail for each:
1. `manifest.yaml` exists and is valid YAML
2. All required fields are present and non-empty
3. Docker image referenced in manifest exists locally
4. Nginx port is not already bound on the host
5. Generated `nginx.conf` is syntactically valid

```bash
./swiftdeploy validate
```

### `deploy`
Starts OPA first, runs pre-deploy policy checks, then brings up the full stack and blocks until health checks pass or 60s timeout.
```bash
./swiftdeploy deploy
```

**Pre-deploy policy checks:**
- Disk free must be above threshold (default: 10GB)
- CPU load must be below threshold (default: 2.0)
- Memory usage must be below threshold (default: 90%)

If any check fails, deployment is blocked with a clear violation message.

### `promote`
Switches deployment mode between `stable` and `canary` with a rolling service restart.
```bash
./swiftdeploy promote canary
./swiftdeploy promote stable
```
- Runs pre-promote policy check before switching to canary (error rate + P99 latency)
- Updates `mode` field in `manifest.yaml` in-place
- Regenerates `docker-compose.yml` with the new `MODE` env var
- Restarts only the app container
- Confirms the new mode is active via `/healthz`

### `teardown`
Removes all containers, networks, and volumes.
```bash
./swiftdeploy teardown
```

Add `--clean` to also delete generated config files:
```bash
./swiftdeploy teardown --clean
```

### `status`
Live-refreshing terminal dashboard that scrapes `/metrics` every 5 seconds.
```bash
./swiftdeploy status
```

Displays:
- Current mode, chaos state, and uptime
- Real-time error rate and P99 latency
- Policy compliance for both infrastructure and canary policies
- Appends every scrape to `history.jsonl` for the audit trail

Press `Ctrl+C` to exit.

### `audit`
Parses `history.jsonl` and generates `audit_report.md`.
```bash
./swiftdeploy audit
```

Report includes:
- Timeline of mode changes and chaos injections
- All policy violations with timestamps
- Summary statistics

## API Endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/` | GET | Welcome message with mode, version, and timestamp |
| `/healthz` | GET | Liveness check — returns status and uptime in seconds |
| `/metrics` | GET | Prometheus metrics in text format |
| `/chaos` | POST | Simulate degraded behaviour (canary mode only) |

### Metrics Exposed

| Metric | Type | Description |
|---|---|---|
| `http_requests_total` | Counter | Total requests labelled by method, path, status_code |
| `http_request_duration_seconds` | Histogram | Request latency distribution |
| `app_uptime_seconds` | Gauge | Seconds since server started |
| `app_mode` | Gauge | 0=stable, 1=canary |
| `chaos_active` | Gauge | 0=none, 1=slow, 2=error |

### Chaos Endpoint (canary mode only)

```bash
# Slow mode — sleep N seconds before responding
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 5}'

# Error mode — return 500 on ~50% of requests
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'

# Recover — cancel any active chaos
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

## OPA Policy Engine

SwiftDeploy uses [Open Policy Agent](https://www.openpolicyagent.org/) as a policy sidecar. All allow/deny decisions live exclusively in OPA — the CLI never makes policy decisions itself.

### Policy Domains

**Infrastructure Policy** (`policies/infrastructure.rego`)
- Blocks deployment if disk free < threshold
- Blocks deployment if CPU load > threshold
- Blocks deployment if memory usage > threshold

**Canary Safety Policy** (`policies/canary.rego`)
- Blocks promotion if error rate > threshold
- Blocks promotion if P99 latency > threshold

### Threshold Configuration

All thresholds are defined in `policies/data.json` — never hardcoded in Rego files:

```json
{
  "infrastructure": {
    "min_disk_free_gb": 10.0,
    "max_cpu_load": 2.0,
    "max_memory_percent": 90.0
  },
  "canary": {
    "max_error_rate": 0.01,
    "max_p99_latency_ms": 500.0
  }
}
```

## manifest.yaml Reference

```yaml
services:
  image: kloudwiz-swiftdeploy:latest  # Docker image name
  port: 3000                           # Internal app port
  mode: stable                         # Deployment mode: stable | canary
  version: 1.0.0                       # App version injected as APP_VERSION
  restart_policy: unless-stopped       # Docker restart policy

nginx:
  image: nginx:latest                  # Nginx image
  port: 8080                           # Host port nginx listens on
  proxy_timeout: 30                    # Upstream timeout in seconds

opa:
  port: 8181                           # OPA API port (localhost only)

network:
  name: swiftdeploy-net                # Docker network name
  driver_type: bridge                  # Network driver

volumes:
  logs: app-logs                       # Named volume for app logs
```
